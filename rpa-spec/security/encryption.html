Memory Encryption

RPA supports hardware memory encryption to protect against physical memory attacks.

Encryption Overview

Memory encryption protects data at rest:

Memory Encryption Flow:
┌──────────────────────────────────────────────────┐
│ CPU                                               │
│   │                                               │
│   │ Plain text                                    │
│   ▼                                               │
│ ┌─────────────────────────────────────────────┐  │
│ │ Encryption Engine                            │  │
│ │   key = domain_encryption_key               │  │
│ │   cipher = AES-XTS(plaintext, key)          │  │
│ └─────────────────────────────────────────────┘  │
│   │                                               │
│   │ Cipher text                                   │
│   ▼                                               │
│ ┌─────────────────────────────────────────────┐  │
│ │ Memory Controller                            │  │
│ └─────────────────────────────────────────────┘  │
│   │                                               │
│   ▼                                               │
│ DRAM (encrypted)                                 │
└──────────────────────────────────────────────────┘

Per-Domain Encryption

Each domain can have its own encryption key:

DCB Field: encryption_key_id
────────────────────────────
Reference to encryption key

Key Management:
┌──────────────────────────────────────────────────┐
│ Key 0: Root domain key                           │
│ Key 1: OS domain key                             │
│ Key 2: Application domain key                    │
│ Key 3: Secure enclave key                        │
└──────────────────────────────────────────────────┘

Hardware automatically:
1. Selects key based on accessing domain
2. Encrypts on write
3. Decrypts on read

Encryption Modes

RPA supports multiple encryption modes:

Encryption Modes

Mode

Description

None

No encryption

Integrity-only

MAC but no encryption

Confidentiality

Encryption without integrity

Confidentiality+Integrity

Full encryption with MAC

 
DCB Field: encryption_mode
──────────────────────────
0 = None
1 = Integrity only
2 = Confidentiality
3 = Confidentiality + Integrity

Memory Bandwidth Impact

Encryption adds latency to memory access:

Latency components:
┌──────────────────────────────────────────────────┐
│ Base DRAM latency:    ~100 cycles               │
│ Encryption overhead:  +~10-20 cycles            │
│ Integrity check:      +~5-10 cycles             │
│ Total:                ~115-130 cycles           │
└──────────────────────────────────────────────────┘

Mitigations:
- Cache encryption keys in on-chip storage
- Use wide-bus encryption engines
- Cache line granularity encryption

Key Management

Encryption keys are managed securely:

Key Lifecycle:

1. Generation
   ─────────────
   Keys generated by hardware RNG
   Never exposed to software

2. Distribution
   ─────────────
   Keys stored in secure key storage
   Referenced by key_id in DCB

3. Rotation
   ──────────
   Parent can rotate child's key
   Requires domain re-encryption

4. Destruction
   ────────────
   Key destroyed on domain termination
   Memory becomes unreadable

Use Case: Confidential Computing

Confidential VM protected from cloud hypervisor:

Domain 0: Cloud Hypervisor (Group 0, Key 0)
  └─ Domain 1: Confidential VM (Group 3, Key 1)

Cloud hypervisor:
- Can manage VM (create, terminate)
- Cannot read VM memory (different security group)
- Cannot decrypt VM memory (different key)

VM owner can verify:
- VM is isolated from hypervisor
- Memory is encrypted
- Via remote attestation
